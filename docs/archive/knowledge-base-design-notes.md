# ナレッジベース設計：追加検討メモ

## この文書の位置づけ

既存設計（登録ツール・検索ツール・Claude自身による苦戦判定・GitHub共有）は、**最小構成としてすでに完成している**。この文書は、運用しながら必要に応じて足していく**追加検討項目**をまとめたもの。いきなり全部やる必要はない。

章立ては優先度順：

- **優先度A**：初期実装に含めると後が楽
- **優先度B**：ある程度運用してから判断
- **優先度C**：共有が活発になってきた段階で考える

---

## 原案の再確認（変更なし）

- **ナレッジの実体**：全プロジェクトファイルのMarkdown
- **登録**：Claudeが苦戦→解決したときに自ら書き込む
- **検索**：Claudeが苦戦したときに自ら調べる
- **共有**：GitHubで管理、皆で集める

この4点はそのまま維持。以下はあくまで「足すなら」の候補。

---

## 優先度A：初期実装に含めたい

### A-1. 登録ツールに `confidence` を入れる

Claudeは「とりあえず動いた」を「解決した」として記録しがちな傾向がある。これを防ぐため、登録時に信頼度を自己申告させる。

```
confidence: "confirmed" | "likely" | "guess"
```

- `confirmed`: 根本原因まで分かっていて、再現性のある解決
- `likely`: 解決したが、因果関係にはまだ仮説がある
- `guess`: 動いたが理由は不明（おまじない状態）

検索時にも `confidence_min` でフィルタできるようにしておくと、ナレッジの質が後々上がる。

### A-2. 最低限のfrontmatterを決めておく

Markdownの自由記述だけで溜めていくと、半年後に検索性と鮮度判定が両方破綻する。最初から**最小限のメタデータ**を必須にする。

```yaml
---
target: "stripe-node"       # 対象ライブラリ/API/ツール名
version: "14.x"             # 該当バージョン（不明なら "unknown"）
symptom: "..."              # 一行で症状
confidence: "confirmed"
encountered_at: "2026-04-18"
---
```

**`version` を入れる理由**：ライブラリは頻繁に更新される。バージョンなしの罠情報は、数ヶ月後に「これまだ該当する？」が判定できず、負債化する。

### A-3. 登録ツールの引数を構造化する

Markdownの本文を生テキストで受け取るのではなく、構造化した引数に分割する。Claudeが思いつきで自由形式に書くのを防ぐ。

```
register_knowledge({
  target: string,          // 必須
  version: string,         // 必須
  symptom: string,         // 必須: 一行で症状
  context: string,         // 必須: どういう状況で踏んだか
  solution: string,        // 必須: 何をして解決したか
  root_cause: string,      // 推奨: なぜ起きていたのか
  confidence: "confirmed"|"likely"|"guess",  // 必須
  keywords: string[]       // 検索用
})
```

ツール側で上記を受け取り、規定のフォーマットでMarkdownファイルに書き出す。Claudeに書式の一貫性を期待するより、ツールで強制する方が確実。

### A-4. 検索ツールは「症状＋リンク」で返す

検索結果に本文全体を含めると、Claudeのコンテキストを消費しすぎる。返却は概要のみにして、詳細はClaudeが必要なら別途ファイルを読む二段構えにする。

```
search_knowledge({ query, target?, confidence_min? })
  ↓
[
  { target, symptom, confidence, file_path, score },
  ...
]
```

Claudeは気になる候補だけ `Read` で本体を読みに行けばいい。

## 優先度B：運用してみて必要なら

### B-1. 統計シグナルによる自動トリガー

Claudeの自己判断に任せると、**「苦戦した自覚がないまま解決したケース」で登録漏れが起きる**。これを補うため、客観的な「苦戦シグナル」を検出して、Claudeに**「今の解決、ナレッジ登録する？」を強制的に問いかける仕組み**を後から足せる。

参考：[Claude Doctor](https://github.com/millionco/claude-doctor) が検出しているシグナル：

- 同一ファイルを1セッションで5回以上編集（edit-thrashing）
- 3回以上連続でツール失敗（error-loop）
- 読む:編集比が10:1以上（excessive-exploration）

プロジェクトに合わせて独自シグナルを定義してもいい。例：

- 同じエラーメッセージが3回以上出現した後に解決
- ユーザーから「違う」「やり直し」系の訂正が2回以上
- テストが何度も赤→最後に緑

これらをHookで検知して、Claudeに登録候補を提案させる。

**いつ導入するか**：最初の1ヶ月運用してログを見て、「登録漏れが多そう」と感じたタイミング。最初からやる必要はない。

### B-2. 鮮度管理

ライブラリがバージョンアップすると、昔記録した罠が嘘になる。以下を足すと安全：

- `last_verified` フィールドをfrontmatterに追加
- GitHub Actionsで定期的に「最終検証から6ヶ月以上経過したナレッジ」を洗い出す
- Claudeが検索でヒットさせるとき、`last_verified` が古ければ警告を付けて返す

**いつ導入するか**：ナレッジが50件を超えて、「もう古い情報かも」と疑いたくなった頃。

### B-3. エピソード層とパターン層の分離

最初は1件1罠で記録していけばいいが、似た罠が複数溜まってきたら、**「エピソード（個別事件）」と「パターン（抽象化した傾向）」を分ける**構造が効いてくる。

- エピソード：`episodes/2026-04-18-stripe-webhook-race.md`
- パターン：`patterns/webhook-signature-timing.md`（複数のエピソードを参照）

パターン層には「この系統の罠にはこういう共通対策がある」という一般化を書く。Claudeがまず引くべきはパターン層、そこからリンクでエピソードに降りる設計。

**いつ導入するか**：同じカテゴリの罠が3件以上溜まったとき。

## 優先度C：共有が活発になってから考える

### C-1. プライバシー/機密の scrub

GitHub共有する場合、**社内プロジェクト名・APIキー・顧客名などが混入するリスク**が必ず出る。登録ツールの内部で、以下をチェック：

- APIキーっぽい文字列（正規表現でマスク）
- 社内固有の識別子（ブラックリスト方式で警告）
- ファイルパスに含まれるユーザー名

最初は「登録前にClaude自身にチェックさせる」指示で十分。共有が活発になったら、pre-commit hookで機械的に弾く仕組みを足す。

### C-2. 多人数運用の信頼性管理

他の人が書いたナレッジをClaudeが引くとき、信頼度をどう扱うか。

- `author` フィールドを足す
- `verified_by` を足して、レビュー通過で `confidence` が上がる運用にする
- 同じ `target` で矛盾するナレッジが出たときの扱い（バージョン違いかも、記録ミスかも）を決める

**いつ導入するか**：共有リポに複数人がコミットし始めて、品質のばらつきが気になった頃。

### C-3. 他プロジェクトとの連携

類似思想のツール・リポと相互運用できる余地を残しておく。直接統合しなくても、スキーマ互換性を意識しておくと後で楽。

- **Claude Doctor**：Claudeの行動癖を検出しルールを生成。守備範囲が違うが相互補完する
- **context7**：公式ドキュメントを引くMCP
- **awesome-claude-code**：Claude Code関連ツール集

## 設計思想の整理

このナレッジベースが目指しているのは、**「ドキュメントには書かれていない罠」を形式知化すること**。公式ドキュメントやStack Overflowでは得られない、**踏んだ人しか知らない情報**を構造化する。

これは以下の2軸で整理できる：

- **Claude Doctor的アプローチ**：Claude自身の行動癖（手続き的知識）
- **このプロジェクトのアプローチ**：世界の仕様の罠（宣言的知識）

両方が揃うと、Claudeは**正しい作法で、正しい情報に基づいて動ける**。

## 注意点

- **最初から全部やろうとしない**：A-1〜A-4は実装負荷が軽いので初期から入れていいが、B以降は「痛みを感じてから」足す方が結果的に綺麗な設計になる
- **Claudeの自己判断を信用しすぎない**：優先度A-1の `confidence` と優先度B-1の統計トリガーは、どちらもClaudeの自己認識の穴を埋めるための仕組み
- **ナレッジは資産であり負債**：溜めれば価値だが、古くなれば毒。鮮度管理（B-2）は思ったより早く必要になる可能性がある
